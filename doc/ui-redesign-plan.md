# QUARCS Stellarium Web Engine UI 重构规划

## 背景

当前前端 UI 采用的是“星图/相机画布 + 覆盖式控制层”的模式。该模式本身适合天文观测、设备控制和拍摄场景，但现阶段控制界面已经承载了大量设备、流程、状态和弹窗逻辑，前端结构明显偏向功能堆叠，后续继续做视觉和交互升级的成本较高。

本次重构的核心思路不是推翻星图引擎，而是：

- 保留星图与画布显示能力
- 重构整个控制界面
- 将星图部分作为新的前端界面的背景层或底层视口
- 在其上构建一套新的控制系统、导航体系和信息层级

一句话概括，即：

保留渲染引擎，重建控制前端。

## 重构目标

- 保留现有 Stellarium 星图和相机画布相关能力
- 重建控制界面的信息架构、视觉系统和交互层次
- 让 UI 从“单体功能堆叠”转向“分层、模块化、可维护”的前端结构
- 降低后续新增设备、模式和面板时的复杂度
- 为未来的多模式体验提供基础，例如：
  - 观测模式
  - 拍摄模式
  - 导星模式
  - 极轴校准模式
  - 设备管理模式

## 当前方案判断

该项目适合采用“星图为背景层，新控制界面为前景层”的分层方案。

这种方式的优点包括：

- 不触碰最复杂、最核心的星图渲染部分，技术风险更低
- 控制界面可以独立重新设计，不被旧布局约束
- 可以逐步替换旧 UI，而不是一次性推翻所有逻辑
- 有利于建立新的设计系统和更清晰的状态管理边界

需要注意的是：

- 新 UI 不应继续直接堆叠在现有大体量 `App.vue` 控制结构上
- 否则会形成“旧 UI 没拆干净，新 UI 又长上去”的双重复杂度

因此推荐策略为：

- 继续复用底层渲染与设备通信能力
- 并行搭建新的 UI Shell
- 在新壳层稳定后逐步替换旧控制界面

## 推荐分层架构

新的前端可以拆为以下 3 层：

### 1. Engine Layer

负责底层渲染和画布能力，尽量保留现有实现：

- Stellarium 星图渲染
- 主相机画布
- 导星相机画布
- 坐标换算
- 目标选择
- 缩放、锁定、跟踪等基础视口能力

这一层的原则是：

- 尽量不重写
- 只做必要的封装与接口清理

### 2. Interaction Layer

负责把底层能力转化为 UI 可以稳定调用的统一接口。

建议由这一层承接的能力包括：

- 模式切换
- 选中目标与取消选中
- GOTO / 锁定 / 缩放
- 设备状态读取
- 拍摄与导星动作触发
- 弹窗和通知分发
- 全局快捷操作
- 业务事件聚合

这一层的目标是：

- 隔离旧逻辑细节
- 降低新 UI 对底层实现的耦合

### 3. UI Shell Layer

这一层是完全重新设计的控制界面。

职责包括：

- 展示状态
- 组织导航
- 呈现面板
- 响应用户操作
- 调用 Interaction Layer 完成业务动作

这一层不再直接负责底层渲染逻辑。

## 建议的新界面结构

可将新的前端界面设计为固定壳层，星图或相机画面作为背景：

### TopBar

用于承载全局与高优先级信息：

- 时间
- 连接状态
- CPU 温度 / 占用
- 当前模式
- 快速搜索
- 全局入口

### LeftPanel

用于承载目标与上下文信息：

- 选中目标信息
- 天体详情
- 坐标与可见性信息
- 目标快捷操作

### RightPanel

用于承载设备与控制面板：

- Mount
- Main Camera
- Guider
- Focuser
- CFW
- Pole Camera

### BottomDock

用于承载高频主操作与模式切换：

- 星图显示开关
- 拍摄
- 导星
- 构图
- 极轴
- 计划任务

### FloatingContextTools

用于承载当前模式下的上下文悬浮控件：

- 极轴模式按钮
- 图像缩放
- 隐藏 UI
- 画面相关快捷操作

### Modal Layer

用于承载低频但需要聚焦的内容：

- 通用设置
- 设备绑定
- 日志
- 位置设置
- 任务详情
- 文件浏览

## 推荐代码组织方式

建议新的前端逐步演进为类似以下结构：

```text
src/
  shell/
    AppShell.vue
    TopBar.vue
    LeftSidebar.vue
    RightSidebar.vue
    BottomDock.vue
    GlobalModalHost.vue

  viewport/
    SkyViewport.vue
    MainCameraViewport.vue
    GuiderViewport.vue
    ViewportHost.vue

  panels/
    target/
    mount/
    camera/
    guider/
    focuser/
    schedule/
    files/
    settings/

  components/
    ui/
      QButton.vue
      QCard.vue
      QStatusBadge.vue
      QToolbar.vue
      QSection.vue
      QDialog.vue
      QTabs.vue

  store/
    modules/
      ui-shell.js
      viewport.js
      devices.js
      session.js
```

这个结构的目标是：

- 将“视口渲染”与“控制界面”解耦
- 将通用 UI 组件和业务面板分离
- 将壳层状态与设备状态拆开

## 对旧模块的处理建议

旧模块建议分为 3 类处理。

### 1. 直接保留

优先保留底层能力与通信链路：

- 星图 canvas 挂载逻辑
- 主相机/导星相机视图切换逻辑
- 底层设备通信与消息机制
- 一部分现有 Vuex 状态结构

### 2. 包装后复用

适合先保留逻辑，再换壳：

- 目标选择相关逻辑
- 设备状态聚合逻辑
- 直方图与调焦数据面板
- 导星数据展示
- 任务计划相关业务逻辑

### 3. 逐步废弃

以下模块更像旧控制壳层，应逐步替换：

- `src/App.vue` 中承担大一统控制界面的部分
- `src/components/gui.vue`
- `src/components/toolbar.vue`
- `src/components/bottom-bar.vue`

核心原则不是立即删除，而是：

- 先复用业务
- 再迁出交互
- 最后退役旧视觉壳层

## 第一阶段最值得落地的内容

建议第一阶段优先做“新骨架”，而不是一开始就重写所有设备面板。

### 阶段 1：新壳层与视口宿主

优先交付：

- `ViewportHost`
- `AppShell`
- 新的基础布局骨架

目标：

- 先把星图和相机视图稳定嵌入新前端框架
- 建立新 UI 的主布局规则

### 阶段 2：高感知区域重做

优先交付：

- `TopBar`
- `RightPanel`
- `BottomDock`

目标：

- 尽快让新界面形成明显可感知的视觉差异
- 替换最常用、最影响观感的控制区域

### 阶段 3：设备与业务面板迁移

逐步迁移：

- Mount 控制
- Camera 控制
- Guider 控制
- Focuser 控制
- CFW 控制
- Schedule 面板

目标：

- 从旧界面中逐步抽离业务操作
- 让新界面成为主工作区

### 阶段 4：清理旧壳层

当新界面已具备主线能力后：

- 关闭旧 UI 入口
- 移除旧控制层依赖
- 清理样式覆盖与遗留布局

## 推荐实施策略

推荐采用并行替换，而不是直接原地改造旧大组件。

建议路线：

1. 保留底层引擎与通信逻辑
2. 新建一套独立的 UI Shell
3. 新壳层先接入现有总线和 store
4. 跑通主流程后逐步替换旧组件
5. 最终切换为新控制界面主入口

这样做的优点：

- 可控
- 容易回退
- 能边开发边验证

## 当前方案的总体结论

本项目不适合继续围绕现有大体量单体 UI 做局部修补式设计升级。

更合适的路线是：

- 保留星图和画布引擎
- 将其视为背景层或底层视口
- 重建新的控制界面外壳
- 通过分层和模块化逐步完成前端替换

这条路线能够同时兼顾：

- 风险控制
- 视觉升级空间
- 交互重构空间
- 长期可维护性
